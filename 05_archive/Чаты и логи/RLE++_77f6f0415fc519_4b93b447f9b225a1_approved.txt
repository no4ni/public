Техническое задание на RLE++
1. Назначение
RLE++ — алгоритм сжатия данных без потерь, ориентированный на:
максимальное сжатие для крайних случаев (полный порядок, полный хаос) даже за счёт ухудшения для средних;
максимальное сжатие для самых частых случаев за счёт остальных.

2. Общие принципы
Входные и выходные данные рассматриваются как последовательности байт.
Сжатое представление состоит из блоков. Каждый блок содержит 4-байтовый заголовок (число N в little‑endian) и паттерн PD переменной длины.
Числа в паттернах кодируются переменным форматом: little‑endian, без ведущих нулей (кроме числа 0, которое представляется одним байтом 0x00).
«Примитивная строка» — строка, не являющаяся степенью никакой более короткой строки (период равен её длине).

3. Правила кодера (порядок применения строго по убыванию приоритета)

	Правило 0 (иерархическое самоподобие)
		Если входные данные D представимы в виде D = S * K + number(K), где: K ≥ 2 (целое); S — некоторая строка; + означает конкатенацию; number(K) — переменное представление числа K; и для некоторого X (1 ≤ X ≤ 4, X ≤ |S|) число, составленное из последних X байт строки S (little-endian), равно K^(a-1) для некоторого целого a ≥ 2, то блок кодируется как:
		[4 байта K^a] + S * a (т.е. заголовок содержит число K^a, паттерн — a повторений строки S).

	Правило 1 (самоподобие)
		Если входные данные D представимы в виде D = P + number(|P|), где number(|P|) — переменное представление длины P (1–4 байта), то блок кодируется как:
		[0x00 0x00 0x00 0x00] + P.

	Правило 2 (вложенное повторение)
		Если D состоит из K одинаковых блоков B, за которыми следует number(K), причём каждый блок B представим как B = Q + number(N) (с одинаковым N), и Q является степенью (Q = S^i, i ≥ 2, S примитивна), то блок кодируется как:
		[4 байта N] + Q.

	Правило 3 (обычное повторение)
		Если D есть повторение примитивного паттерна P ровно N раз (N ≥ 2), то блок кодируется как:
		[4 байта N] + P.

	Правило 4 (примитивные данные)
		Если ни одно из правил 0–3 не применимо, то D считается примитивным и кодируется как:
		[0x01 0x00 0x00 0x00] + D.

4. Правила декодера
Декодер получает блок: первые 4 байта — число N (little-endian), остальное — паттерн PD.

	А. Если N == 0
		Вывести PD, затем number(|PD|) (переменное представление длины PD).

	Б. Если N > 0
		Проверить, является ли PD степенью (PD = S^i, i ≥ 2, S примитивна).

		Если да: а)
			Пусть a = i (показатель степени).
			Найти целое K ≥ 2 такое, что K^a == N. 
				I. Если такое K существует, то это случай, соответствующий правилу кодера 0. 
					Вывести K раз строку S, затем number(K).
				II. Если K не найдено, то применяется старое правило (соответствующее правилу кодера 2):
					Вычислить K:
						–1- если N == 1, то K = a;
						–2- если N > 1, то K = N^(a-1).

						Вывести K раз пару (S, number(N)).
						Затем вывести number(K).

		Если нет (PD примитивна): б)
			Вывести PD ровно N раз (правило кодера 3 или 4).

5. Примечания
number(x) — переменное представление числа x (little‑endian, минимум байт, без ведущих нулей; для x=0 используется один байт 0x00).
Под «степенью» понимается строка, которая может быть представлена как минимум два повторения некоторой примитивной строки. Для проверки используется поиск наименьшего периода.
Возведение в степень выполняется целочисленно; результат должен умещаться в 32 бита (иначе правило не применяется).
Приоритет правил кодера жёстко фиксирован. Декодер однозначно восстанавливает исходные данные благодаря проверке условий.

7. Критерии приёмки
Кодер для любого корректного входного блока выдаёт допустимый сжатый блок.
Декодер для любого сжатого блока восстанавливает исходные данные.
Все примеры из п. 6 обрабатываются без ошибок.
Внутренняя непротиворечивость правил (нет конфликтов, каждый вход кодируется единственным способом).

8. Ограничения
Максимальное значение N в заголовке — 0xFFFFFFFF.
Переменное представление чисел ограничено 4 байтами, поэтому при переполнении берём просто последние 4 байта (даже с ведущими нулями)
При поиске степени K^a == N перебор ограничен разумными пределами (до N).
Вложенность блоков не превышает глубины, обусловленной размером стека вызовов (в реализациях следует предусмотреть защиту от бесконечной рекурсии).

9. Перспективные расширения (экспериментальные)
	1. В будущих версиях может быть введено правило R, основанное на рекурсивном декодировании. Идея состоит в следующем: если входные данные D можно представить как результат N-кратного применения декодера к некоторому блоку O (т.е. D = decode^N(O)), то блок кодируется как [N] + O. Однако практическая реализация требует эффективного алгоритма поиска таких O и N, что на данный момент не реализовано и остаётся открытым вопросом.

	Пример (иллюстративный, не рабочий в текущей версии):
	Пусть D = [05 00 00 00 01]. Тогда:

	decode(D) = [01 01 01 01 01] (пять единиц).

	Если теперь интерпретировать результат как новый блок [01 01 01 01 01], то его первые 4 байта — 01 01 01 01 = 0x01010101 (16843009), а паттерн — [01]. Тогда decode от этого блока даст [01], повторённый 16843009 раз, что не равно D. Таким образом, данное представление не удовлетворяет условию D = decode^2(O), но демонстрирует саму идею многократного применения декодера.

	Включение такого правила потребует дальнейших исследований и, возможно, новых подходов к поиску рекурсивных структур в данных.

	2. Симметричное самоподобие (число в начале)
	Нет правила для данных вида number(|P|) + P. Такая структура потенциально может быть закодирована компактнее, но текущая система использует для этого другие правила (например, если P примитивно, то попадёт в правило 4, что даст заголовок 0x01 и само P — но это не учитывает наличие числа в начале). Введение отдельного правила с новым значением заголовка (например, N = 2) потребовало бы пересмотра занятых значений, что не было сделано.

	3. Блоки с N = 0 и PD, не являющимся примитивным
	В текущем декодере при N == 0 мы просто выводим PD и number(|PD|), независимо от того, является ли PD степенью. Это означает, что блоки с N = 0 и паттерном-степенью никогда не генерируются кодером, но если бы они появились (например, в результате будущих расширений), декодер обработал бы их по правилу А, что не всегда соответствует исходным данным. Это можно рассматривать как резервное пространство для новых команд.

	4. Ограничение на длину числа (1–4 байта)
	Правила самоподобия (правило 1) и иерархическое правило 0 используют number(|P|) и number(K), которые ограничены 4 байтами. Это значит, что для очень длинных паттернов (длина > 2³²−1) такие правила неприменимы.

	Рекомендация на будущее: переход к битовому кванту
		В текущей версии RLE++ минимальной единицей данных является байт. Однако для достижения ещё более высокого сжатия в НЕКОТОРЫХ сценариях может быть оправдан переход к битовому кванту (требуется дополнительное число определяющее размерность кванта - бит/байт/два байта/четыре) — когда все операции (поиск периодов, кодирование чисел, представление паттернов) выполняются на уровне отдельных битов. Это позволит более точно упаковывать данные, особенно если в них встречаются короткие последовательности, не выровненные по байтам, или если часто используются маленькие числа.

		Потенциальные преимущества
		Более плотное сжатие для данных с высокой избыточностью на битовом уровне (например, монохромные изображения, битовые карты, некоторые форматы телеметрии).

		Возможность кодировать числа с переменным количеством бит, отбрасывая ведущие нули справа (аналог number(x), но для битов), что даёт дополнительную экономию для малых значений.

		Устранение «байтового выравнивания» там, где оно не нужно.

		Технические соображения
		Программная реализация: Для ускорения операций с битами можно использовать приёмы, при которых данные временно дополняются нулями до ближайшего байта (или машинного слова) при выполнении вычислений, но в итоговом сжатом потоке хранятся только значащие биты. Это позволяет сочетать эффективность побитовых алгоритмов с быстродействием байтовых операций процессора.

		Кодирование чисел: Числа в паттернах и заголовках могут представляться переменным числом бит, где первым битом указывается продолжение (как в UTF-8, но для битов), либо просто хранится минимальное количество бит без ведущих нулей (с права, так как little-endian). Например, число 5 (биты 101) можно сохранить как три бита 101, а не как целый байт.

		Поиск периодов: Алгоритмы поиска повторяющихся паттернов должны быть адаптированы для работы с битовыми строками, что усложняет реализацию, но принципиально возможно.

		Ограничения и риски
		Сложность реализации: Переход на битовый уровень требует переработки практически всех компонентов кодера и декодера, включая проверку на степень, поиск периодов, работу с переменными числами.

		Скорость работы: Побитовые операции медленнее байтовых, хотя описанный приём с дополнением нулями может частично компенсировать это.

		Совместимость: Изменение формата сжатого потока сделает его несовместимым с предыдущими версиями. Это может быть приемлемо только при создании новой ветки алгоритма.

		Заключение
		Переход на битовый квант является перспективным направлением для будущих исследований и оптимизаций RLE++. Он позволит достичь теоретического предела сжатия в тех областях, где байтовая гранулярность избыточна.
