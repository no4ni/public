Техническое задание на систему TreeDiscus
1. Общее описание
TreeDiscus — это платформа для построения и анализа древовидных аргументационных карт с глубокой интеграцией источников.
Основное назначение: помочь сообществу исследователей, философов, журналистов и всех заинтересованных структурировать дискуссии, выявлять противоречия, оценивать достоверность утверждений на основе проверяемых данных.

Ключевое отличие от аналогов (Kialo, Arguman, Rationale):

Источники — объекты первого класса. Каждое утверждение может сопровождаться библиографической ссылкой, документом или выдержкой.

Автоматический анализ источников: проверка на взаимные противоречия внутри одного документа, выделение ключевых тезисов.

Динамическая репутация участников: вес голоса зависит от вклада, подтверждённого сообществом, и давности регистрации.

Многоязычный интерфейс и контент с возможностью переключения между языками.

Два режима отображения: общая карта (коллективное знание) и персональная ветка (моё личное мнение).

2. Цели проекта
Создать инструмент для доказательного обсуждения сложных тем.

Снизить влияние субъективных предпочтений за счёт привязки к источникам и их автоматического анализа.

Обеспечить прозрачную эволюцию знаний через историю правок и голосований.

Дать возможность каждому пользователю вести собственную версию карты аргументов, не разрушая общую картину.

3. Функциональные требования (MVP)
3.1. Управление дискуссиями
Создание новой темы (корневой узел).

Древовидное добавление узлов двух типов:
Утверждение (pro/contra) и Категория (группировка).

Перемещение узлов в иерархии (drag-and-drop в интерфейсе).

Версионирование правок (кто, когда, что изменил).

Модерация: возможность пометить узел как «недостоверный» / «нарушающий правила».

3.2. Работа с источниками
К узлу можно прикрепить один или несколько источников:

URL-ссылка,

загруженный файл (PDF, DOCX, TXT, изображение),

вручную введённая цитата с выходными данными.

Автоматический парсинг:

Извлечение текста,

Распознавание языка,

Разбиение на отдельные утверждения-сегменты.

Анализ противоречий (на основе NLP/встроенной логики):

Внутри одного источника: поиск фраз, противоречащих друг другу.

Между источниками, привязанными к одному узлу: подсветка конфликтующих позиций.

Отчёт по источнику: процент совместимости с текущей картой дискуссии, ключевые слова.

3.3. Голосования и репутация
Каждый зарегистрированный участник может голосовать:

за/против истинности утверждения (вес ±1),

за релевантность источника (0..1),

за качество категоризации (попадание узла в нужную ветку).

Вес голоса рассчитывается как:

Базовый вес = 1.

Бонус за подтверждённые добавленные источники (количество одобренных правок × коэффициент).

Бонус/штраф за время регистрации (медленно растущий логарифмический коэффициент).

Итоговая оценка узла — взвешенная сумма с учётом весов голосовавших.

Возможность просмотра истории своих голосов и вклада.

3.4. Режимы отображения
Общая карта (Global Tree): коллективное знание, усреднённое по голосам.

Личная ветка (My View): пользователь может «перетащить» узлы в свою версию, скрыть чужие аргументы, создать персональную структуру без влияния на общую.

Экспорт карты в JSON, HTML, PNG.

3.5. Поиск и фильтрация
Поиск по тексту узлов, тегам, источникам.

Фильтр по:

дате добавления,

рейтингу достоверности (высокий/средний/низкий),

языку,

наличию/отсутствию источника,

автору правки.

3.6. Мультиязычность
Интерфейс минимум на русском и английском (переключается без перезагрузки).

Контент: автоматическое определение языка, возможность добавить ручной перевод узла на другие языки.

Кросс-языковой поиск (морфологический анализ для поддерживаемых языков).

4. Архитектура системы
4.1. Клиент-серверное взаимодействие
Серверная часть: REST API (или GraphQL) на Python (FastAPI) или C# (.NET 6+).

Клиент: SPA на React / Vue.js с визуализацией дерева через D3.js или Cytoscape.js.

База данных: PostgreSQL (основные данные), Elasticsearch (полнотекстовый поиск + логи).

Хранилище файлов: локальная файловая система или S3-совместимое хранилище, с антивирусной проверкой при загрузке.

4.2. Ключевые алгоритмы/сервисы
Анализатор противоречий (Contradiction Detector) — отдельный микросервис, может использовать предобученные модели типа Sentence-BERT для сравнения утверждений.

Репутационный движок — вычисление весов в реальном времени при голосовании (можно фоном пересчитывать раз в час).

Индексатор источников — извлечение текста, разбиение на сегменты, определение языка.

5. Интерфейс пользователя (ключевые экраны)
Рабочий стол (Dashboard): список доступных дискуссий, фильтры, строка поиска.

Карта дискуссии: интерактивное дерево с раскрывающимися узлами, панель деталей (утверждение, источники, голоса, кнопки действий).

Редактор узла: добавление/редактирование текста, прикрепление источников (URL, файл, цитата).

Отчёт по источнику: модальное окно с автоматически найденными противоречиями, облаком слов.

Профиль участника: вклад, голоса, история правок.

6. Этапы реализации (дорожная карта)
Этап 0 – Прототип локального сервера + минимальный клиент (уже начат в папке FormalTesis)
Создание БД и API для базовых операций CRUD узлов.

Простейший веб-клиент для отрисовки дерева без голосования.

Этап 1 – MVP с основными функциями
Регистрация, аутентификация.

Добавление узлов, категорий, источников (URL, файлы).

Голосование с простым весом (равный для всех).

Поиск и фильтрация.

Локализация интерфейса (ru/en).

Этап 2 – Анализ источников и репутация
Парсинг загруженных PDF/DOCX.

Детектор противоречий v1 (на основе ключевых слов и метрик).

Внедрение репутационных весов.

Этап 3 – Продвинутые функции
Детальный анализ источников (ML), кросс-языковое сравнение.

Личное дерево (My View).

Экспорт, отчёты, API для внешних сервисов.

7. Технические детали и ограничения
Требования к серверу: на начальном этапе — обычный VPS (2 ядра, 4 ГБ RAM). Для анализа PDF/ML может потребоваться GPU.

Безопасность: HTTPS, шифрование паролей, защита от XSS/CSRF, проверка загружаемых файлов.

Масштабируемость: возможность горизонтального масштабирования Elasticsearch и базы данных.

8. Интеграция с другими твоими проектами
Возможно использовать FormalTesis как модуль формальной логики, связывая утверждения через правила вывода.

Экспорт итоговой карты в Википедию или другие базы знаний.

Привязка к GitHub-репозиторию для версионирования текстов источников.
